# Device-Aware Banking Authentication Platform вҖ” Spec & Action Plan

## Context

The user wants to build a multi-component platform combining identity management, device enrollment, and banking access control. The core requirement is that a user can only access the banking application from a device they have previously enrolled via a Device Enrollment Service (DES). WSO2 IS enforces this during login via an adaptive authentication script that calls the DES.

**Key constraints confirmed:**
- Device ID = UUID generated by DES at enrollment time; user copies it into the banking login form
- User registration = WSO2 IS built-in self-service portal (no custom registration page needed)
- DES backend = **already written in Ballerina** вҖ” integrate, do not rebuild
- Banking frontend = React SPA (to be built)
- WSO2 IS = configuration + adaptive script (to be set up)

## 4. Database Schema (`database/des.sql`)

MySQL database: `device_enrollment`

### `devices` вҖ” enrolled device records

| Column | Type | Notes |
|---|---|---|
| `device_id` | VARCHAR(255) PK | UUID generated by DES at enrollment |
| `device_name` | VARCHAR(255) | User-supplied label |
| `device_type` | VARCHAR(255) | e.g. `"Mobile"` or `"PC"` |
| `user_name` | VARCHAR(255) | WSO2 IS username |
| `created_at` | TIMESTAMP | Auto-set on insert |
| `updated_at` | TIMESTAMP | Auto-updated on change |

### `device_users` вҖ” userвҶ”device authorization & session state

| Column | Type | Notes |
|---|---|---|
| `id` | INT AUTO_INCREMENT PK | |
| `device_id` | VARCHAR(255) FK вҶ’ devices | CASCADE on delete |
| `user_id` | VARCHAR(255) | WSO2 IS username |
| `is_authorized` | BOOLEAN | Default TRUE |
| `is_active` | BOOLEAN | TRUE while user has an active login session |
| `created_at` | TIMESTAMP | |

**Unique constraint:** `(device_id, user_id)`

**Indexes:** `idx_device_id`, `idx_user_id`, `idx_user_active (user_id, is_active)`

## 5. User Flow (End-to-End)

### 5.1 Registration
1. User navigates to WSO2 IS self-service portal (`https://localhost:30003/accountrecoveryendpoint/register.do`)
2. Fills in name, email, password вҶ’ account created in WSO2 IS

### 5.2 Device Enrollment
1. User signs in to the **des-ui** device manager (`http://localhost:5173`) via WSO2 IS OIDC
2. In the dashboard, enters a device name and clicks **Mobile** or **PC**
3. `des-ui` calls `POST http://localhost:8080/device` with `{Id, deviceName, deviceType, userName}`
4. DES generates a UUID, stores the device in `devices` + `device_users` tables, returns `{id: deviceId}`
5. Enrolled device appears in the device list; user copies the UUID for banking login

### 5.3 Banking Login (Device-Validated)
1. User visits the banking app login page
2. Triggers OIDC Authorization Code + PKCE flow, appending `device_id` as a custom query parameter
3. WSO2 IS processes credentials, then fires the adaptive script
4. Adaptive script extracts `device_id`, calls `GET /device/verify/{device_id}/{user_id}` on DES
5. DES checks authorization, enforces `maxLoginCount=3` concurrent sessions вҶ’ responds success or error
6. WSO2 IS allows login (issues tokens) or rejects with error
7. On success, DES is notified via `POST /device/authenticated/{device_id}/{user_id}` to set `is_active=true`
8. On logout, DES is notified via `POST /device/logout/{device_id}/{user_id}` to set `is_active=false`

## 6. WSO2 IS Adaptive Authentication Script

File: `identity/adaptive-script.js` вң…

The script uses the **actual DES verify endpoint** вҖ” a path-param GET, not a query-param URL:

```
GET http://des-service:8080/device/verify/{device_id}/{user_id}
```

The script:
1. Runs Step 1 (username + password)
2. Extracts `device_id` from `context.request.params`
3. Calls `GET /device/verify/{device_id}/{username}` via `httpGet`
4. On HTTP 200 вҶ’ login proceeds (tokens issued)
5. On any error / non-200 вҶ’ `sendError` with appropriate error code

**Error codes defined in the script:** `DEVICE_ID_MISSING`, `DES_UNREACHABLE`, `DEVICE_NOT_PERMITTED`, `DES_INVALID_RESPONSE`

> **Note:** IS 6.x/7.x uses `httpGet`. See `identity/adaptive-script.js` for the full implementation.

## 8. WSO2 IS Configuration Steps (Summary)

Full step-by-step instructions are in `identity/README.md`.

1. Start WSO2 IS вҶ’ access admin console at `https://localhost:30003/carbon`
2. Register **BankingApp** as OIDC Service Provider (Auth Code + PKCE, callback `http://localhost:3000/callback`)
3. Enable Adaptive Authentication on the SP вҶ’ paste `adaptive-script.js`
4. Set `DES_BASE_URL` and `DES_INTERNAL_API_KEY` values in the script
5. Whitelist DES endpoint in `deployment.toml` for outbound HTTP
6. Verify self-service user registration is enabled

---

## 9. Security Considerations

1. `device_id` validated server-side in the adaptive script вҖ” client cannot bypass it
2. DES `/devices/validate` endpoint is internal-only, protected by API key
3. User-facing DES endpoints (`/devices/enroll`, etc.) require a valid WSO2 Bearer token
4. PKCE prevents authorization code interception attacks
5. Device policy limit (max N devices per user) enforced server-side in DES
6. HTTPS required in production; docker-compose uses HTTP for local dev only

| Delete device via des-ui | Row removed from `devices` + `device_users` |
| Exceed `maxLoginCount=3` active sessions | DES returns error, no new active session |
| Login with enrolled `device_id` | WSO2 issues tokens; `is_active` set to TRUE |
| Logout | `is_active` set to FALSE via logout endpoint |
| Login with unknown `device_id` | Adaptive script rejects вҖ” `DEVICE_NOT_PERMITTED` |
| Login with no `device_id` | Adaptive script rejects вҖ” `DEVICE_ID_MISSING` |
